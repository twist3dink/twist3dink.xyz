Understanding Card Information

When you read a card, the Flipper displays several important pieces of information:

  • UID (Unique Identifier): The card's serial number, usually 4 or 7 bytes
  • Card Type: MIFARE Classic, NTAG, EM4100, etc.
  • Capacity: For memory cards, how much data they can store
  • SAK/ATQA: Technical identifiers that reveal card capabilities
  • Sectors: For MIFARE Classic, the number of data sectors

‚ö†Ô∏è What You Can and Can't Read

Can Read:

  • Card UID and basic information
  • Card type and manufacturer data
  • Unencrypted memory areas
  • Public data on transit cards

Cannot Read:

  • Encrypted sectors without keys
  • Payment card account numbers (CVV, etc.)
  • DESFire secure applications
  • Protected authentication tokens

Cloning and Emulation

Once you've read a card, you can clone it to a writable card or emulate it directly from your Flipper. This is useful for legitimate purposes like backing up your access cards or testing system vulnerabilities.

Cloning to Physical Cards

To create a physical copy, you'll need blank writable cards:

  • For 125 kHz: T5577 writable cards (cheap, ~$1 each)
  • For 13.56 MHz: MIFARE Classic writable cards or NTAG tags
  • Purchase from electronics suppliers or online retailers
  • Verify compatibility with your reader system

Cloning Process (125 kHz)

  1. Read the source card and save it
  2. Navigate to 125 kHz RFID ‚Üí Saved
  3. Select your saved card
  4. Choose Write
  5. Place a blank T5577 card on the Flipper
  6. Confirm the write operation
  7. Test the cloned card with the original reader

Cloning Process (13.56 MHz MIFARE Classic)

  1. Read the card (may require key attack for encrypted sectors)
  2. Navigate to NFC ‚Üí Saved
  3. Select your saved card
  4. Choose Write
  5. Place a blank MIFARE Classic card on the Flipper
  6. Wait for the write to complete (takes longer than 125 kHz)
  7. Verify the clone works

üí° UID vs Full Clone

There are two types of clones:

  • UID Clone: Copies only the card ID. Works on systems that only check the UID.
  • Full Clone: Copies UID and all accessible data. Required for systems that read card contents.

Simple access systems often only check the UID, making UID clones sufficient. More sophisticated systems require full clones.

Direct Emulation (No Physical Card Needed)

One of the Flipper's most powerful features is the ability to emulate cards directly without creating a physical clone:

  1. Read and save a card
  2. Navigate to Saved in the appropriate menu
  3. Select your card
  4. Choose Emulate
  5. Hold the Flipper up to the reader like a card
  6. The Flipper transmits the card data wirelessly

‚úÖ Emulation Advantages

  • No need to buy blank cards
  • Store unlimited cards on one device
  • Quick switching between different cards
  • Can't lose or damage the card
  • Perfect for testing and demonstrations

‚ö†Ô∏è Emulation Limitations

  • Some readers detect emulation and reject it
  • Requires the Flipper to be powered on

MIFARE Classic Security Testing

MIFARE Classic is one of the most widely deployed contactless card technologies, but it has known cryptographic weaknesses. Understanding these vulnerabilities is essential for security professionals.

MIFARE Classic Structure

MIFARE Classic cards are organized into sectors and blocks:

  • 1K cards: 16 sectors, each with 4 blocks (64 blocks total)
  • 4K cards: 40 sectors with varying block counts (256 blocks total)
  • Each block stores 16 bytes of data
  • Each sector is protected by two keys: Key A and Key B
  • Keys control read/write access to the sector

Default Keys

Many MIFARE Classic deployments use default or common keys. The Flipper has a dictionary of known keys:

  • FFFFFFFFFFFF (most common default)
  • A0A1A2A3A4A5 (transport configuration)
  • 000000000000 (sometimes used)
  • Manufacturer-specific defaults
  • Industry-specific common keys

Dictionary Attack

The first attack to try against MIFARE Classic cards:

  1. Navigate to NFC ‚Üí Read Card
  2. Place the MIFARE Classic card on the Flipper
  3. Flipper automatically attempts dictionary attack
  4. Wait for the attack to complete (usually seconds)
  5. If successful, you'll see which sectors were unlocked
  6. Save the card with recovered keys

üí° Dictionary Attack Success Rate

Dictionary attacks are surprisingly effective:

  • ~30-40% of deployed systems use default keys
  • Hotel key cards often use predictable keys
  • Legacy transit systems frequently vulnerable
  • Corporate access systems sometimes have default keys on low-security sectors

Nested Attack (Advanced)

If the dictionary attack doesn't unlock all sectors, the nested attack uses known keys to crack unknown ones:

  • Requires at least one known key from dictionary attack
  • Exploits cryptographic weakness in MIFARE authentication
  • Can recover all keys if at least one is known
  • Takes longer than dictionary attack (several minutes)

Hardnested Attack (Most Advanced)

When no keys are known, the hardnested attack can still succeed:

  • Works even when all keys are non-default
  • Requires significant computation time
  • Usually performed on a computer, not Flipper
  • Demonstrates that MIFARE Classic fundamentally insecure

‚ö†Ô∏è The Bottom Line on MIFARE Classic

MIFARE Classic should not be used for high-security applications.

  • The encryption algorithm (Crypto-1) is broken
  • Keys can be recovered with sufficient time and tools
  • Widely considered obsolete for security-critical systems
  • Many organizations continue using it due to cost and inertia

When assessing systems, recommend upgrading to MIFARE DESFire or other secure alternatives.

Conducting Card Security Assessments

When authorized to assess an organization's physical access control, follow this methodology:

Assessment Process

  1. Reconnaissance
    • Identify card types in use (visual inspection, vendor info)
    • Determine reader types and models
    • Understand the access control architecture
    • Document card distribution and management processes
  2. Technical Testing
    • Read issued cards to identify technology
    • Test for default or weak keys (MIFARE Classic)
    • Attempt cloning and emulation
    • Test reader response to cloned cards
    • Assess UID-only vs full-authentication systems
  3. Range Testing
    • Test maximum read distance for attacks
    • Assess feasibility of wallet skimming
    • Document environmental factors affecting range
  4. Process Review
    • Evaluate card issuance procedures
    • Review card revocation processes
    • Assess physical security of card stock
    • Check for regular security updates
  5. Reporting
    • Document all vulnerabilities found
    • Provide risk ratings and impact assessments
    • Recommend specific remediation steps
    • Suggest timeline for improvements

Common Vulnerabilities to Test For

UID-Only Authentication

System only checks the card UID, not encrypted data.

Risk: High - trivial to clone

Test: Clone UID to blank card and test access

Default Encryption Keys

MIFARE Classic using factory-default keys.

Risk: High - keys easily recovered

Test: Dictionary attack reveals default keys

No Encryption

125 kHz cards with no security features.

Risk: Critical - anyone can clone

Test: Simple read and write to T5577

Weak Key Diversity

All cards in deployment use same keys.

Risk: High - one compromise affects all

Test: Compare keys across multiple cards

No Card Revocation

Lost/stolen cards not properly deactivated.

Risk: Medium - extended exposure window

Test: Check if old cards still work

Physical Security Gaps

Readers accessible for tampering or relay attacks.

Risk: Varies by location

Test: Physical inspection of readers

Security Best Practices

When you identify vulnerabilities, provide concrete recommendations for improvement:

Immediate Improvements

  • Change default keys: If using MIFARE Classic, program unique keys
  • Enable full authentication: Check card data, not just UID
  • Implement card revocation: Deactivate lost/stolen cards promptly
  • Physical reader security: Install readers in tamper-resistant enclosures
  • Regular audits: Periodically test card security

Long-Term Upgrades

  • Migrate to secure card technology: MIFARE DESFire, iClass SE, or similar
  • Implement multi-factor: Combine cards with PIN codes or biometrics
  • Use mobile credentials: Leverage smartphone-based access
  • Network-based management: Real-time card activation/deactivation
  • Logging and monitoring: Track all access attempts

‚úÖ Recommended Card Technologies

For new deployments or upgrades, recommend:

  • MIFARE DESFire EV2/EV3: Industry standard, strong encryption
  • HID iClass SE: Secure, widely supported
  • Mobile credentials: Smartphone-based, highly secure
  • Smart cards with PKI: For highest security requirements

Avoid: EM4100, HID Prox (125 kHz), MIFARE Classic for new deployments.

Advanced Topics

For security researchers wanting to go deeper, here are advanced concepts to explore:

Relay Attacks

Relay attacks extend the range of contactless cards by relaying communication between a legitimate card and a reader:

  • Attacker places reader near victim's card (in wallet)
  • Signal is relayed to accomplice at target location
  • Accomplice gains access using relayed signal
  • Difficult to detect and prevent
  • Mitigation: Distance-bounding protocols, shielded wallets

UID Modification

Some writable cards allow UID modification (called "magic" cards):

  • Can set any desired UID
  • Useful for testing UID-based systems
  • Some systems detect magic cards
  • Available as "Chinese magic cards" or "Gen1/Gen2" cards

Analyzing Custom Protocols

Organizations sometimes implement proprietary card formats:

  • Requires reverse engineering card data structure
  • Look for patterns in data across multiple cards
  • Identify facility codes, user IDs, checksums
  • Test variations to understand validation logic

Understanding Contactless Technology

Contactless cards and badges are everywhere‚Äîoffice access cards, hotel keys, transit passes, payment cards, and more. The Flipper Zero can interact with these technologies for security testing and research purposes.

This lesson focuses on two main technologies: RFID (Radio Frequency Identification) and NFC (Near Field Communication). While related, they serve different purposes and have distinct security characteristics.

‚ö†Ô∏è Critical Legal Warning

Card cloning and unauthorized access are serious crimes.

  • Never clone or use payment cards (credit/debit)‚Äîthis is fraud
  • Only test access cards you own or have explicit authorization to test
  • Don't gain unauthorized entry to buildings or restricted areas
  • Respect privacy‚Äîdon't scan cards in other people's wallets
  • Some countries criminalize possession of cloning devices

This lesson is for security professionals learning to test and improve access control systems they're authorized to assess.

NFC vs RFID: Key Differences

While both use radio frequencies for communication, they're designed for different applications.

Feature Low Frequency RFID (125 kHz) High Frequency NFC (13.56 MHz)
Frequency 125 kHz 13.56 MHz
Read Range Up to 10cm typical Up to 10cm (intentionally limited)
Data Rate Very slow 106-424 kbps
Typical Uses Access cards, key fobs, animal tags Payment cards, transit cards, smartphones
Security Usually none (just a static ID) Can include encryption and authentication
Complexity Simple, easy to clone More complex, better security

üí° Quick Rule of Thumb

  • 125 kHz RFID: Older technology, simpler, less secure. Common in legacy access systems.
  • 13.56 MHz NFC: Modern technology, more features, better security. Used in payment cards and smartphones.

Common Card Types

Understanding what type of card you're dealing with is the first step in security assessment. Here are the most common types you'll encounter:

Card Technology Breakdown

EM4100 (125 kHz)

Security: None (read-only ID)

Common in:

  • Older access control systems
  • Key fobs for buildings
  • Some gym memberships

Easily cloned! Just contains a static ID number.

HID Prox (125 kHz)

Security: Minimal (proprietary format)

Common in:

  • Corporate access systems
  • Government facilities
  • Parking garages

Can be cloned with proper tools and knowledge.

MIFARE Classic (13.56 MHz)

Security: Weak encryption (broken)

Common in:

  • Transit systems worldwide
  • Hotel room keys
  • Access control systems

Encryption is crackable with known attacks.

MIFARE DESFire (13.56 MHz)

Security: Strong (AES, 3DES)

Common in:

  • High-security access systems
  • Government ID cards
  • Secure transit systems

Secure when properly implemented.

EMV Payment Cards (13.56 MHz)

Security: Very strong (tokenization)

Common in:

  • Credit and debit cards
  • Contactless payment systems

Cannot be cloned for fraudulent use. Don't try!

NTAG (13.56 MHz)

Security: Varies (can be locked)

Common in:

  • NFC tags and stickers
  • Smart posters
  • Product authentication

Often writable for legitimate applications.
